## 브레인스토밍 합의 사항

### 재정비 성격

**C. 복합** — 핵심 철학(Performance > Everything, B>P>A>D>C, AI-native) 유지.
현실(4가지 drift)을 직시하고, 표현을 갱신하며, 측정 가능한 게이트 추가.

---

### § 1 가설 — 합의 내용

**3단 논리**:
1. 기존 언어는 인간 인식 한계 위에 설계 → 모호성·런타임 오버헤드 구조적 잔존
2. AI가 코드를 작성하면 언어가 어려워도 무관 → 완전한 성능 지향 처음으로 가능
3. AI도 기계어 불가 (컨텍스트 한계, 구조 파악, 속도, 비용) → AI가 다룰 수 있는 가장 낮은 추상화 수준 = BMB

**성능 목표 = 존재가치 증명**:
- LLVM 백엔드 등 외부 요인 제외, 언어 한계·컴파일러 한계로 포기 불가
- 문제 발견 시 언어 스펙·컴파일러·부트스트랩 어느 레벨이든 low level부터 개선 게이트 항상 열려있어야 함

**도그푸딩 = 존재가치 증명의 메커니즘**:
- 컴파일러·부트스트랩·에코시스템·외부 패키지 작업 전부 도그푸딩 활동의 일환
- 도그푸딩 중 발견된 문제는 workaround 금지, low-level 게이트로 처리

---

### § 2 진단 — 4가지 Drift

| Drift | 현상 | 근본 원인 |
|-------|------|---------|
| A | 생태계 작업이 성능 목표와 별개로 보임 | 도그푸딩 프레임 미명시 |
| B | B가 #1인데 한 번도 측정 안 함 | Track R "추적만, 합격선 없음" |
| C | AI-native 선언, 그러나 LLM 자연 패턴 미지원 | 언어 갭 해소가 작업 항목 미등록 |
| D | CLAUDE.md·docs/ROADMAP.md·vision spec 각각 독립 진화 | 유일한 앵커 문서 없음 |

---

### § 3 처방

- **A**: 모든 활동 도그푸딩으로 분류, 문제 발견 시 low-level 게이트
- **B**: Track R에서 LLM 1-shot 성공률 baseline 측정 (M4 첫 액션)
- **C**: LLM 미지원 패턴 목록화 → 언어 스펙 개선 백로그
- **D**: `claudedocs/ROADMAP.md` = 유일한 실무 앵커로 지정

---

### § 4 로드맵 구조

```
현재: M1 ✅ → M2 ✅ → M3 🔄(75%) → M4 ⬜ → v1.0 ⬜

M3 잔여: showcase 선정(HUMAN) + 벤치 측정 + npm/PyPI publish
M4 핵심: B baseline 측정 + 언어 갭 해소 + 추가 바인딩
v1.0 조건: 외부 신호 충족 (비자율)
```
