## 2. 정체성 (Identity)

### 2.1 BMB는 무엇인가

> **"AI 시대의 컴파일러·언어 도구를 위한, 가장 낮은 추상화 수준의 contract-verified systems language"**

이 한 문장이 향후 모든 결정의 앵커다.

### 2.2 1차 사용자 = 인간+AI 협업

- **인간**: 자연어로 의도를 전달 (LLM에게)
- **LLM (Claude/GPT 등)**: BMB 코드 작성 — 외부 도구, BMB 컴파일러와 통합 X
- **BMB 컴파일러**: contract로 검증, 결정론적 IR 생성, 구조화 출력

**함의**:
- 친절한 인간 에러 메시지보다 **기계 검증 가능한 에러 구조**가 우선
- AI에게 별도 채널/합성기 X — BMB 언어 자체가 AI-friendly하게 설계됨이 본질
- *"AI가 BMB를 잘 다루지 못한다"는 외부 변수* — 우리 통제 영역에서는 언어 설계와 주변 도구(MCP/CLI)로 충분히 함

### 2.3 1차 도메인 = 컴파일러·언어 도구·DSL·검증기

**왜 이 도메인인가**:
- BMB는 이미 BMB로 자기 자신을 컴파일한다 (`bootstrap/compiler.bmb` 32K LOC) — 도메인이 코드에 박혀있음
- Contract 검증의 가치가 가장 명확한 영역 (parser invariants, type checker soundness, IR rewriter correctness)
- AI가 가장 잘 생성하는 영역 (정형적·알고리즘적·트리 변환)
- LLVM 백엔드 강점과 직접 정합

**도메인 외 작업의 처리**:
- mandelbrot, n-body 등 수치 계산 벤치마크: **강등** (도메인 외)
- brainfuck, lexer, hash_table: **유지** (도메인 정합)
- 8/15 FAIL의 다수가 도메인 외 — 새 우선순위에서 자원 회수 (정확한 카운팅은 메타 정렬 사이클에서 분류표 작성)

## 3. 우선순위 (Priority Order)

### 3.1 AI-Ready 5축 — B > P > A > D > C

| 순위 | 축 | 측정 |
|------|----|------|
| 1 | **B (Failure Rate)** | LLM이 spec → BMB 1-shot 컴파일+verifier 통과율 |
| 2 | **P (Performance)** | C/Rust 등가 알고리즘 대비 (특히 컴파일러 도메인) |
| 3 | **A (Token Efficiency)** | 동일 알고리즘 LOC·토큰 수 비교 (BMB vs Rust vs Python) |
| 4 | **D (Verification Coverage)** | contract 자동 증명률, runtime check 제거율 |
| 5 | **C (Context Navigability)** | LLM이 N-파일 프로젝트에서 정답률 |

**의존 관계**: B = 0이 아니면 다른 모든 축 무의미. P/A는 B 위에서. D/C는 장기 차별화.

### 3.2 메타 원칙

- **"Performance > Everything"의 진짜 의미**: "Performance is the only thing"이 아니라 "**일단 동작하고(B), 그 위에서 Performance가 1순위**"
- **외부 변수(AI 학습 수준, 채택률)는 개발 게이트가 될 수 없음** — 우리 통제 영역에서 충실
- **외부 영향 극복 = CLI/MCP 등 주변 도구** — AI가 BMB 학습 못한 상황을 도구로 보완 (컨텍스트 주입, 문법 안내)

## 4. 마일스톤 (Milestones, 자율 게이트)

각 마일스톤은 우리 통제 영역의 binary 조건으로 정의된다. 외부 변수는 게이트가 되지 않는다.

### 4.1 M1: Self-Validated (내부 자기검증 완성)

| 조건 | 측정 |
|------|------|
| Bootstrap Fixed Point | S2 == S3 (현재 ✅) |
| G.1 verifier 결함 fix | Track G의 P1-P3 시퀀스 완수 (prelude duplicate clamp 제거) |
| 컴파일러 도메인 벤치마크 | brainfuck/lexer/hash_table 등 C 동등-추월 |

### 4.2 M2: AI-Ready Infrastructure (Q6 = AI 친화 인프라)

5개 신규 트랙으로 정의:

| 트랙 | 내용 | 비고 |
|------|------|------|
| **M (Machine-First Output)** | 모든 컴파일러 출력 기본 = 구조화 JSON, `--human` 옵션 | **부분 구현 — `--human` 플래그 이미 존재** (`docs/superpowers/specs/2026-03-25-ai-friendly-tooling-design.md`). 잔여 작업: 모든 명령(`bmb verify`, `bmb bench` 등)에 일관 적용 + 스키마 안정화 |
| **N (MCP Server)** | `bmb mcp` 명령, BMB 문법/타입/spec 리소스 노출 (Claude Desktop/Code 등 통합) | 신규 |
| **O (Context Pack)** | `bmb context-pack <project>` — AI 토큰 효율적 프로젝트 요약 (모듈 트리, 공개 인터페이스, 주요 contract) | 신규 |
| **Q (Ambiguity Audit)** | grammar 정적 분석 + `bmb lint --ai-friendly` — 모호 파스 가능 위치 0 | 신규 |
| **R (LLM Bench Tracking)** | `bmb llm-bench` + 50개 표준 task suite — **추적만, 합격선 X** | 신규 |

### 4.3 M3: External Bindings PoC (외부 바인딩 첫 검증)

| 조건 | 측정 |
|------|------|
| BMB 라이브러리 (BMB 특징을 드러내는) 1개 | 후보: lexer 생성기, IR 변환기, JSON parser 등 (M3 진입 시 결정) |

| 트랙 S 90% | LSP, fmt, lint, verify, bench가 BMB로 작성됨 |

### 4.4 M4: Adopted (광범위 바인딩 + 외부 채택 신호 충족)

| 조건 | 측정 |

## 6. 버전 정책 (Release Versioning, 비자율 게이트)

**원칙**: 마일스톤 도달과 메이저 버전 선언은 **다른 차원**이다.

| 차원 | 결정 권한 | 게이트 |
|------|---------|------|
| 기술 마일스톤 (M1~M4) | 자율 (자가 검증) | 내부 binary 조건만 |
| 릴리스 버전 (v0.x → v1.0 → v2.0) | **비자율** (메인테이너 + 커뮤니티) | **외부 신호 필수** |

### 6.1 현재 → v1.0 사이의 명명

현재 v0.98. 마일스톤 진척에 따라 마이너 버전만 증가. v1.0은 외부 신호 충족 시 별도 선언.

| 마일스톤 도달 | 권장 버전 |
|--------------|---------|
| M1 도달 | v0.99 (내부 도달 시) |
| M2 도달 | v0.100 / v0.110 |
| M3 도달 | v0.150 / v0.200 — **v1.0 후보 — 외부 신호 평가 시작** |
| M4 도달 | **v1.0 선언** (외부 신호 충족 시) |

### 6.2 v1.0 외부 신호 임계값 (가-합의, M3 진입 시 정식 확정)

| 신호 | 임계값 | 측정 |
|------|------|------|
| GitHub stars | ≥ 1,000 | 자동 |
| 외부 PR merged | ≥ 10 (각각 다른 contributor) | 자동 |
| 외부 이슈 (월) | ≥ 10 (활성 사용자 신호) | 자동 |
| 부정 평가 비율 | < 30% (HN/Reddit/social 노출 후) | 수동 |
| 외부 BMB 프로젝트 | ≥ 5 (BMB로 작성된 외부 프로젝트) | 수동 |
| 결정자 | 메인테이너 + 외부 contributor 협의 | 정책 |

**왜 이렇게 분리하는가**: v1.0은 "안정적이며 사용자가 의지할 수 있다"는 **약속**이다. 외부 사용자 없이 v1.0을 일방 선언하면 그 약속의 의미 자체가 무너진다 (Zig가 v0.x인 이유, Rust가 v1.0까지 7년 걸린 이유).

## 9. 비전 한 문단 요약 (Pitch)

> BMB는 컴파일러·DSL·검증기를 작성하는 인간+AI 협업 팀을 위한 contract-verified systems language이다. AI가 BMB로 코드를 작성하면 컴파일러가 결정론적으로 IR을 생성하고 contract로 검증한다. 별도 AI 채널 없이 언어 자체가 AI-friendly하다 — 모호성 없는 문법, 토큰 효율 표현, 구조화 출력이 1급 시민이다. 외부 LLM의 BMB 학습 수준은 우리 통제 밖이지만, MCP 서버·context-pack 같은 주변 도구로 보완한다. 자기 자신을 BMB로 검증하고 (M1), AI 친화 인프라를 갖추고 (M2), 외부 언어에 바인딩으로 BMB 라이브러리를 노출하고 (M3), 외부 채택이 자연 따라온 후 (M4) v1.0을 선언한다.
