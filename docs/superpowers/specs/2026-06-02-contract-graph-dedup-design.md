# 계약 그래프 기반 중복 제거 — 설계 문서

> **날짜**: 2026-06-02
> **상태**: 설계 승인 (구현 계획 대기)
> **범위**: Phase 1 (A 토대) + Phase 2 (A 강제 + B 합성). 부채꼴 소비자는 플랜 종료 후.

## 0. 배경 — 왜 이 작업인가

### 0.1 두 가설의 "동등" 진단

BMB는 두 헤드라인 가설로 출발했다: **최고성능** + **AI-Native**. 그러나 정직한 측정은 두 축 모두에서 "동등"으로 수렴했다:

- **성능**: BMB는 LLVM을 백엔드로 쓴다. Clang도 같다. **같은 백엔드를 공유하는 한 성능 상한은 Clang이다.** AL 포렌식 감사(Cycles 3527-3559)는 "7/7 faster than Clang"이 거짓이며 실제는 vs-Clang 3 소폭우위/3 파리티/1 열위 = **파리티**임을 확인했다. "Zero Overhead"는 "C 능가"가 아니라 "최적 C와 동급"의 다른 이름이며, 이는 이미 Rust의 약속이다.
- **AI-Native**: B축 98~100%는 "LLM이 5회 안에 통과 프로그램을 만든다"를 측정할 뿐, "BMB가 Python/Rust보다 쉽다"를 측정하지 않는다. CLAUDE.md의 friction 목록(`bshr` vs `>>`, `new` 강제, hashmap i64::MIN 등)은 오히려 gotcha가 더 많음을 보여준다.

**핵심 통찰**: 두 가설이 *하나로 묶이는* 유일한 지점은 — AI가 쓴 계약 → 증명 → 옵티마이저가 동등 C에서 못 얻는 정보 → C가 못 만드는 IR. 이 피드백 루프가 작동해야 두 가설이 동시에 참이 된다. 현재 데이터는 이 루프가 측정 가능한 형태로 작동함을 보여주지 못한다.

### 0.2 진짜 문제 — 대형 프로젝트 바이브코딩의 구조적 붕괴

`claudedocs/research/AI 네이티브 언어 설계 요구사항 도출.md`는 바이브코딩의 7대 페인포인트(A~G)를 실증 분류한다. 주류 언어(Python/Rust/TS)는 이를 구조적으로 다루지 못한다. BMB의 점수표 ("충족"=기능 출하, ≠페인 측정):

| 요구사항 | 대응 페인 | BMB 현황 |
|---------|----------|---------|
| 6 정적 capability/effect | F 보안 | ✅ 실질충족 (하드 보장) |
| 3 인라인 SMT 자가치유 | C 검증 | ✅ 실질충족 (레이턴시 미검증) |
| 2 intent↔contract | B 명세갭 | ⚠️ 부분 (선택적) |
| 4 영속 아키텍처 상태 | D 맥락손실 | ⚠️ 부분 |
| 7 결정론적 레이아웃/락 | G 재현성 | ⚠️ 부분 |
| **1 의미론 중복 감지** | **A 중복** | ❌ **약함** (호출집합 휴리스틱) |
| 5 추상화 억제 | E 유지보수 | ❌ 약함 |

**아이러니**: 사용자가 BMB 자체(167K LOC)를 만들며 실제로 겪는 페인이 A(중복)인데, A가 BMB의 최약점이다. 현 `SemanticDuplicate`(M14)는 "두 함수가 같은 헬퍼를 부르는가"의 호출집합 휴리스틱일 뿐, 계약도 시그니처도 보지 않는다.

### 0.3 돌파구

> **함수의 정체성은 이름도 시그니처도 아니라 계약이다.** 계약(`pre`/`post`)은 인간 언어에 없는, *무엇을 하는지*에 대한 기계 검증 가능 명세다. (pre, post)가 Z3로 등가인 두 함수는 같은 함수다.

이것이 요구사항 1을 **임베딩 유사도(휴리스틱)가 아니라 정리 증명(하드 보장)**으로 푸는 길이며, BMB의 최강 자산(계약)으로 최약 페인(중복)을 친다.

## 1. 핵심 — 계약 정체성과 레지스트리

### 1.1 정체성 키

```
identity_key = (정규화된 입력 타입들, 출력 타입, 정규형 완전-post)
```
함수의 이름이 아니라 계약이 정체성.

### 1.2 완전성 판정 (Z3)

post `Q(x, it)`가 출력을 유일하게 결정하는가?
```
∀x. pre(x) → ∃!it. Q(x, it)
   ≡ ∀x,it₁,it₂. pre(x) ∧ Q(x,it₁) ∧ Q(x,it₂) → it₁ == it₂
```
- 증명됨 → **완전 계약**: 레지스트리 1급 시민
- 아니면 → **부분 계약**: 경고 전용으로 격하

### 1.3 등가 판정 (Z3)

같은 타입 시그니처 + 둘 다 완전 +
```
∀x. pre(x) → (Q_a(x,it) ⟺ Q_b(x,it))
```
증명되면 둘 다 완전하므로 **관측 등가가 증명됨** → 구현이 달라도 안전하게 같은 함수.

### 1.4 레지스트리 (`.bmb-registry`, 디스크 영속)

`identity_key → { canonical_name, location, complete: bool }`. 디스크에 전역 의미 그래프를 통째로 보존 — 인간 컨텍스트 한계 언어가 못 보는 것을 컴파일러/AI가 본다. "고정관념 탈피"의 물리적 구현체.

### 1.5 정직한 경계

`∀`/`∃!` 질의는 특히 String 이론·quantifier에서 미결정·느릴 수 있다. **분류 불가한 계약은 자동으로 "부분/경고"로 폴백** (건전성 우선). 폴백 비율은 §4 측정 대상.

## 2. dedup-check 발화 흐름 (Phase 1: 작성 시점 권고)

**새 명령**: `bmb dedup-check <file>` — `semantic-duplicate` 형제로 추가, diagnose JSON 배관 재사용.

```
파일 파싱 → 각 함수 계약 추출
  → 완전성 Z3 질의 (∃!it)
     ├─ 완전: 레지스트리 등가(⟺) 질의
     │    ├─ 등가 존재 → contract_duplicate (증명됨)
     │    └─ 없음      → 레지스트리 등록 (새 canonical)
     └─ 부분/미결정: 호출집합 휴리스틱 → potential_duplicate (경고)
```

**출력 — 증명/휴리스틱 절대 분리** (advisor 규율):
```json
{ "type": "contract_duplicate", "function": "make_secure_token",
  "equivalent_to": "AuthUtil.create_secure_string",
  "proof": "z3_equiv", "signature": "(i64) -> String",
  "action": "redirect", "confidence": 1.0 }
```
```json
{ "type": "potential_duplicate", "function": "...", "candidates": [...],
  "basis": "call_set_overlap", "confidence": 0.7 }
```

**MCP 래핑**: `bmb_dedup_check` 툴 → LLM이 함수 *생성 전* 질의 → "이미 등가 존재, 리디렉션" 피드백. 사용자 실 페인을 작성 루프 안에서 직격.

**레지스트리 적재**: `bmb dedup-check --register <file>`로 기존 코드 스캔 → 완전-계약 함수 적재.

**언어/컴파일러 변경 0** — 순수 도구 계층. 측정 후 Phase 2로 승격.

## 3. Z3 배관 — 옵션 B (스코프 한정)

현 경로: `smt/translator.rs`(BMB→SMT-LIB2, M7 String 포함) + `smt/solver.rs`(`z3 -in` 프로세스에 stdin SMT2 주입). solver는 임의 SMT2를 받는 범용 경로.

**결정: compiler.bmb가 SMT2를 직접 방출 + z3 셸 호출 (Rule 6 준수).**

"compiler.bmb엔 SMT 번역기가 없다 → 거대 작업"이라는 반론은 §1.5 경계에서 사라진다: **완전성을 주장하는 건 i64/bool 산술·비교 단편뿐**이고, String/quantifier는 이미 "부분/경고"로 폴백한다. 그래서:

- compiler.bmb는 **결정 가능한 산술/bool 단편만 SMT2로 직렬화** (`+`/`-`/`*`/비교/`and`/`or`)
- z3에 던지고, 번역 불가는 호출집합 휴리스틱 경고로 폴백
- 번역기 표면이 작고, soundness 경계와 정확히 일치

산술/bool 단편이 dedup 기회의 대부분을 잡는지는 §4 측정으로 검증. 못 잡으면 translator 확장을 Phase 2 결정으로.

**필요 인프라**: BMB의 `str_sb`(SMT2 빌드) + `exec_with_stdin`(z3 호출) — 둘 다 M6에 존재.

## 4. 측정 하니스 (Phase 1의 진짜 산출물)

**대상**: BMB 자신 — `compiler.bmb` 48K → 전체 bootstrap 167K. 사용자 실 페인이 시험대.

| 지표 | 정의 | 의미 |
|------|------|------|
| 완전성률 | 계약 함수 중 ∃!it 증명 비율 (산술/bool 단편) | 건전 경로 커버리지 |
| dedup 기회 | ⟺ 등가 클러스터(≥2 멤버) 수 | 페인이 측정으로 나타나는가 |
| 폴백률 | String/quantifier로 격하 비율 | 단편 한정의 실제 한계 |
| Z3 레이턴시 | 질의당 + 48K 전체 wall-clock | 자가치유 루프 레이턴시 위험 시험 |

**O(n²) 폭발 방어 (설계 일부)**:
1. **시그니처 버킷** — `(입력타입, 출력타입)` 같은 함수끼리만 비교
2. 버킷 내 **구문 정규형 선통과** — 자명 동일은 Z3 전에 제거
3. 그 다음만 Z3 ⟺
4. `verify/incremental.rs`·`proof_db.rs` 증명 캐싱 재사용 — 같은 쌍 재질의 0비용

**정직한 결과 시나리오 (셋 다 유효 산출물)**:
- dedup 기회 多 + 레이턴시 수용 → thesis 증거. Phase 2로.
- 완전성률 低 → "AI가 완전 계약을 쓴다"가 가정이 아니라 **선행 과제**임이 측정으로 드러남. B의 전제 조건 발견.
- 레이턴시 高 → Req 3 위험 실재. 캐싱+버킷팅 완화 가능성이 다음 질문.

**진다 해도 그게 정직한 첫 측정**이며, BMB가 새 축(대형 프로젝트 붕괴 저항)에서 자신을 잰 최초 데이터.

## 5. B로 가는 길 + 부채꼴 수용 설계

### 5.1 핵심 원칙: 레지스트리를 *그래프*로 설계 (Phase 1부터)

```
node = { identity_key, canonical_name, location,
         signature, pre, post, effect_row, complete: bool }
```
dedup은 노드 룩업만 쓰지만, 스키마는 **엣지를 수용할 여지를 예약** → B와 부채꼴이 마이그레이션 없이 꽂힌다. A가 막다른 길이 아니라 토대가 되는 지점.

### 5.2 B (합성) — 이 플랜 스코프, 그래프 위에서

노드를 계약-엣지 `{P}→{Q}`로 보면, 목표 `{P_t}...{Q_t}` 합성 = **그래프 경로 탐색** (각 접합부에서 Z3가 `post_i ⟹ pre_{i+1}` 검증). 시그니처 타입 + 깊이 한계로 탐색 제한. **같은 레지스트리 소비** — A를 만들면 B 탐색 공간이 공짜로 생김.

### 5.3 플랜 단계

- **Phase 1 — A 토대**: 그래프 스키마 레지스트리 + Z3 완전성/등가 + `bmb dedup-check`(작성 시점) + BMB 자기 코드 측정. (§1~4)
- **Phase 2 — A 강제 + B (Phase 1 측정으로 게이팅)**:
  - 2a: 컴파일 시점 정규화 (A → 하드 보장 + 바이너리 크기 측정) — 레이턴시 통과 시
  - 2b: 그래프 경로 탐색 합성 (B) — 완전성률 충분 시
- **플랜 종료 후 — 부채꼴 (비구현, 스키마만 수용)**: 분석(아키텍처 메모리)/성능(스펙 폴딩)/리팩터-안전성/자가치유/결정성.

### 5.4 정직 고지

B의 경로 탐색은 야심 영역 — 중간 술어 무한으로 일반적으로 어렵다. Phase 1 완전성률이 낮으면 B 탐색 공간이 빈약하다는 뜻 → Phase 2b는 go/no-go 게이팅.

## 부록 A — 계약 활용 레버 맵 (부채꼴 전체)

> 중복은 *목표*가 아니라 증명 기반 계약 그래프의 *첫 소비자*다. 아래 전부 같은 그래프의 소비자.
> 표기: `[있음]` 부분구현 / `[그래프 후 저렴]` 증분 / `[야심]` 연구급

**중복/적게작성**: 계약-정체성 dedup (A)`[설계중]` · 계약 분해·합성 (B)`[야심]` · 스펙→구현 합성`[그래프 후 저렴]`
**크기**: dedup→구현 1회 방출`[그래프 후 저렴]` · 합성→코드 아닌 호출 경로`[야심]` · 계약-유도 dead-branch`[있음]`
**성능**: 스펙 수준 상수 폴딩(실행 없이 접음)`[그래프 후 중간]` · 전역 그래프 기반 최적 inline/outline`[그래프 후 중간]` · 안전한 전역 메모이즈/CSE`[그래프 후 저렴]`
**안정**: Z3 verify`[있음]` · 리팩터 안전성(행동 보존 증명, Rust 산사태 직격)`[그래프 후 중간]` · 호출 경계 계약 일관성`[그래프 후 중간]` · SAT-witness 반례`[있음]`
**분석**: 계약=body 안 읽는 기계 요약`[그래프 후 저렴]` · 레지스트리=영속 아키텍처 메모리(Req 4)`[그래프 후 저렴]` · 계약 diff=의미 changelog`[그래프 후 저렴]`
**???**: 계약-고정 자가치유 루프(완전 계약=불변 오라클, 수렴)`[그래프 후 중간]` · 결정성(완전 계약→관측 출력 동일)`[그래프의 따름정리]`

## 부록 C — 미해결 위험

1. **Z3 레이턴시 vs 자가치유 루프** (연구문서 미해결질문 #1): 48K에서 분 단위면 Req 3이 규모에서 무너짐. 버킷+정규형+캐싱으로 완화 — §4 측정으로 검증.
2. **완전성률 미지**: 산술/bool 단편이 실제 계약의 큰 몫인지 미측정. 낮으면 A의 건전 경로가 좁고 B 전제가 약함.
3. **부분 계약 불건전성**: 자동 리디렉션은 완전 계약에만. 부분은 경고만 — 출력에서 절대 혼동 금지.
