## Scope & Implementation

### 새 pure_cse 벤치마크 생성
- `contract_opt/pure_cse/bmb/main.bmb`: @pure @noinline 10중 중복 호출 테스트
- `contract_opt/pure_cse/c/main.c`: 동일 알고리즘 (noinline, recursive)

### 핵심 발견: C 컴파일러의 순수성 분석 능력

**발견 1**: GCC/Clang은 `__attribute__((noinline))` 함수에서도 순수성 분석 수행
- 10중 호출 → 3개로 축소 (both GCC and Clang)
- 재귀 함수도 분석 가능

**발견 2**: BMB의 실제 성능 우위 메커니즘 = **Phase Ordering**
- BMB: MIR PureFunctionCSE (인라인 전) → LLVM 인라인
- C: LLVM 인라인 → GVN/CSE (인라인 후 패턴 인식 실패)
- purity_opt 2.88x 우위의 진짜 원인: BMB가 인라인 전에 CSE 수행

**발견 3**: @noinline + @pure는 별도 컴파일 단위의 정확한 시뮬레이션이 아님
- 현대 C 컴파일러도 noinline 함수의 순수성 분석 가능
- 진짜 별도 컴파일 단위(LTO 없음)에서만 BMB @pure의 절대적 우위 발현

### 성능 메커니즘 정리

| 시나리오 | BMB 우위 | 원인 |
|----------|---------|------|
| **재귀 순수 함수 (같은 파일)** | **2.88x FASTER** | Phase ordering: MIR CSE before LLVM inline |
| 단순 루프 순수 함수 (noinline) | ~1.0x | 현대 C 컴파일러도 순수성 감지 가능 |
| 별도 컴파일 단위 (LTO 없음) | **이론적 ∞** | C는 body 볼 수 없어 CSE 불가, BMB는 @pure로 가능 |
