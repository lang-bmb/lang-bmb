## Scope & Implementation

### Cycle 321: Auto-growing Vec<i64> via realloc

* **`stdlib/core/vec.bmb`**: added `vecg_reserve<T>`,
  `vec_push_i64` with auto-grow, `vec_free<T>` → `vecg_free<T>`.
* **`bmb/src/interp/eval.rs`**: rewrote `builtin_malloc` /
  `builtin_realloc` / `builtin_free` / `builtin_calloc` to call
  libc directly via `extern "C"`. The old `std::alloc::alloc` path
  could not implement `realloc` correctly because it needed the
  original `Layout`; the interpreter now actually copies old bytes
  on grow, matching native semantics.
* **Codegen fix — `StructInit` narrow-source widening**: the text
  codegen was emitting `store i32 %cap, ptr %..._f2` into an
  `i64` struct field, leaving the high 32 bits undefined. Fixed by
  looking up the declared field type from `struct_defs` and
  emitting `sext i32 → i64` when the source has been narrowed.
  This was the root cause of the "cap = 519691042818" garbage
  observed during the first grow test. Before the fix, the bug was
  latent because no stdlib code had ever widened a narrowed
  parameter through a `new Struct { ... }` initializer.
* **Global rename**: all unused-generic vec helpers are prefixed
  `vecg_` (vecg_new, vecg_with_capacity, vecg_len, vecg_cap,
  vecg_clear, vecg_free, vecg_reserve) to dodge symbol collisions
  with the legacy `vec_new()` / `vec_free(i64)` runtime
  intrinsics declared unconditionally by the text codegen. Keeping
  those declarations works because no unmonomorphized generic body
  ever lands on the same symbol.

## Review & Resolution

During the review phase we hit several latent defects that had to
be resolved in-cycle rather than carried forward:

1. **Interpreter realloc leaked old data.** The pre-Cycle-321
   implementation allocated fresh memory but never copied the old
   bytes, so `vec_push` after a grow returned uninitialized
   garbage. Fixed by routing through `libc::realloc` (which the
   native path was already using).

2. **Struct-field stores narrowed integers silently.** The small-
   int narrowing optimization turned `let v = vecg_with_capacity(
   0, 2)` into a call with `i32` arguments; when the constructor
   stored that i32 into the `i64` `cap` field, the upper 32 bits
   were left uninitialized. The sret copy then propagated the
   garbage to every field-read that followed. Root-cause fix: the
   text codegen now sign-extends narrow sources to the declared
   field type at every `StructInit` store.

3. **`load_f64` was missing from the return-type table.** The
   result temp was typed `i64`, then the function's
   `ret double %temp` produced a clang type error. Fixed by
   adding `load_f64 → double`.

4. **Unused generic stdlib functions collide with runtime
   declarations.** When a generic function like `vec_new<T>` is
   declared but never monomorphized, the text codegen emits it
   with the raw generic name, which collides with the
   unconditionally-declared `@vec_new()` runtime intrinsic. Short
   fix: rename the stdlib generics to a `vecg_` prefix. A proper
   fix (skip emission of unmonomorphized bodies OR skip runtime
   declarations shadowed by user functions) is deferred.

5. **Codegen `getelementptr %struct., ...` regression.** An
   earlier revision of `vec_push_i64` threaded a local `grown`
   out of a merge block, and the text codegen emitted
   `getelementptr inbounds %struct., ...` with an empty struct
   type name. Refactoring the grow path to keep `grown` inside the
   then-branch sidestepped the emission bug. Root cause left as
   a follow-up defect.

6. **Alloca-in-loop miscompile.** The sret return slot for
   `vec_push_i64` was being allocated at the call site, inside
   main's `while` body. Combined with defect (2), this produced
   incorrect cap readings at runtime. The struct-field fix
   covered the observable symptom, but the alloca-in-loop pattern
   is still suboptimal (stack grows per iteration until function
   return) and should eventually be hoisted to the entry block.

## Carry-Forward

### Pending Human Decisions

* Should the `vec_new` / `vec_free` / `vec_len` runtime
  intrinsics be renamed (e.g. `bmb_vec_new`) so stdlib can own
  the short names? Would break bootstrap compiler programs that
  hand-call the legacy vec runtime.
* Should `core::hashmap` be rewritten as a true `HashMapG<K, V>`
  on top of `VecG<T>` now, or after turbofish syntax lands so
  callers don't need boilerplate type annotations?
* Version bump target for this cycle's work (v0.97.4 or v0.98
  alpha)? Left unbumped pending human call.

### Discovered out-of-scope

* **Text codegen: `getelementptr %struct., ...` regression** on
  certain phi-merge receivers. Sidestepped in stdlib by inlining
  the grow path; root cause unfixed.
* **Text codegen: `alloca` emitted inside loop bodies** instead
  of the entry block for sret call slots. Currently relies on
  LLVM's `StackAllocator` to do the right thing.
* **Text codegen: unmonomorphized generic bodies** can collide
  with runtime declarations.
* **Interpreter struct-by-value vs native struct-by-ptr** —
  functions that mutate field-by-field in a specialization work
  on native but silently no-op on interpreter unless the caller
  reassigns the return value. Every stdlib helper now follows
  the "reassign the return" convention consistently to bridge
  the gap, but the divergence remains a language correctness
  concern.
* **Vec<T> for non-8-byte T** (i32, u8, etc.). Needs generic
  `store<T>` / `load<T>` intrinsics that dispatch on `sizeof<T>`.
* **Binary_trees benchmark migration.** Original sprint goal —
  deferred because the codegen fixes consumed the budget.

### Next Recommendation

Cycles 341+ should:

1. Fix the text-codegen **alloca-in-loop** pattern by hoisting
   all sret call slots to the entry block. This unblocks more
   natural `mut v = f(v, ...)` loops in stdlib code.
2. Fix **unmonomorphized-generic emission** so `vec_new<T>` can
   live alongside the `vec_new()` runtime intrinsic without the
   `vecg_` prefix hack.
3. Land **turbofish call syntax** (`vec_new::<i64>()`) so the
   seed-argument workaround can retire.
4. Migrate **`binary_trees`** benchmark to `VecG<i64>` and
   measure vs raw-pointer baseline — the sprint's original Phase C.
5. Consider removing the `vecg_clear<T>` generic if it turns out
   any caller benefits more from a monomorphized specialization
   (it does nothing except set a field; the generic form is only
   useful because its monomorphization is never emitted unless
   used).
