# Cycle 3232: S2 Bootstrap IR Truncation Fix (post it >= 0 → -1)
Date: 2026-05-28

## Re-plan
Plan valid. Root cause of S2 IR truncation was identified in the previous session:
`ifs_check_flex_both_sides` had `post it >= 0` but returns `-1` (= `0 - 1`) on all failure paths.
This cycle: fix the contract violation + remove all DBG diagnostic code + verify Fixed Point.

## Scope & Implementation

### Root Cause
`ifs_check_flex_both_sides` (bootstrap/compiler.bmb ~line 25551) declared `post it >= 0`
but returned `0 - 1` = -1 in ALL failure cases. The bootstrap compiler's Return Range
Attribute system (v0.96.30) converts `post it >= 0` → LLVM `range(i64 0, ...)` attribute
on the function return value, AND injects `llvm.assume(ret >= 0)`.

When S1 compiled compiler.bmb to produce S2, LLVM saw `ifs_check_flex_both_sides` was
declared to always return ≥ 0. In caller `ifs_try_two_or_fallback`, the check
`if flex >= 0` was always true, causing LLVM to eliminate the
`else { ifs_emit_branch_fallback(...) }` branch as dead code.
Result: S2 never called `ifs_emit_branch_fallback`, truncating MIR output.

### Fix Applied
**File: bootstrap/compiler.bmb**
1. `ifs_check_flex_both_sides`: `post it >= 0` → `post it >= -1`
   (matches actual behavior: returns -1 for "no match", ≥ 0 for "matched")

2. Removed all DBG diagnostic prints added during diagnosis:
   - DBG1 in `gen_program_acc_sb_structs_reuse`: `[DBG] fn=... mir_len=... branch=YES/NO`
   - DBG8 in `optimize_if_select_twice`: `[IFS-1st] total_len=...`
   - DBG9/10 in `ifs_function`: `[IFF-in]`, `[IFF-out]`, content dumps
   - DBG in `optimize_cf_dce_loop`: 12x `[CF-*]` sub-pass diagnostics + `[CF-ifs-RAW]`
   - DBG in `compile_program`: `[P0-raw]`, `[P1-ci]`, `[P2-cf]`, `[P3-trl]`, `[P4-licm]`

**File: bmb/src/build/mod.rs** (already done in previous session)
   - Both link locations: 64MB → 256MB stack (`-Wl,--stack,268435456`)

## Reflection
- **Scope fit**: Exact fix for the root cause (1-char change: `>= 0` → `>= -1`).
- **Philosophy**: This is a proper fix, not a workaround. The contract was wrong, not the function logic.
- **Completeness**: All diagnostic code removed, no production output pollution.
- **Structural insight**: The bootstrap compiler's `post it >= N` mechanism (Return Range Attribute + llvm.assume) is powerful but creates a meta-circular hazard — a wrong post-condition in S1 can make S2 incorrect by LLVM dead-code elimination of unreachable branches. This is a category of bugs unique to self-hosting compilers.
- **User-facing**: No API changes. Fixed Point confirmed.
