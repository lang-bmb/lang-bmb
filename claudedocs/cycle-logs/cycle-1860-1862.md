# Cycles 1860-1862: MIR Store-Load Forwarding + Constant Folding Expansion
Date: 2026-03-11

## Inherited → Addressed
From 1858-1859: tower_of_hanoi 1.15x FAIL (assembly identical to C), max_consecutive_ones FAIL.

## Scope & Implementation

### 1. MIR Store-Load Forwarding (Cycle 1860)
Enhanced `MemoryLoadCSE` pass with Store-Load Forwarding:
- `store_i64(ptr, val)` → subsequent `load_i64(ptr)` → forwards `val` directly
- `store_f64(ptr, val)` → subsequent `load_f64(ptr)` → forwards `val` directly
- `FieldStore { base, field, value }` → subsequent `FieldAccess { base, field }` → forwards `value`
- Added `operand_to_inst()` helper to generate `Copy` (for Place) or `Const` (for Constant)
- Changed cache value types from `Place` to `Operand` for unified handling
- Updated 3 tests to match new Store-Load Forwarding behavior

### 2. Bitwise & Float Constant Folding (Cycle 1861)
Added missing constant folding patterns to `fold_binop()`:
- **Bitwise**: `Band(const, const)`, `Bor(const, const)`, `Bxor(const, const)`
- **Shifts**: `Shl(const, const)`, `Shr(const, const)` with bounds check (0 ≤ shift < 64)
- **Float comparisons**: `FEq`, `FNe`, `FLt`, `FLe`, `FGt`, `FGe` for constant f64

### 3. tower_of_hanoi Investigation (Cycle 1862)
- Binary .text reduced from 52K to 8.8K (gc-sections working)
- C binary .text: 6.5K — difference is MinGW CRT + import stubs
- Assembly comparison shows different LLVM unrolling/register choices (not identical anymore)
- Previous "identical assembly" conclusion was before gc-sections fix changed binary layout
- LLVM generates structurally similar but not byte-identical code from BMB vs C IR
