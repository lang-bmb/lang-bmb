### 해결 방안 분석
1. **타입드 포인터** (`*i64` + `p[idx]`): GEP inbounds i64 생성 → 벡터화 우호적

2. **load_i64/store_i64 GEP 개선**: 현재 패턴 매칭 기반 → 복잡한 offset 패턴 일부 누락
3. **수용**: 동일 LLVM 백엔드에서 10% 차이는 허용 범위

### 결론
- floyd_warshall 10% 차이는 `load_i64`/`store_i64` 사용 패턴의 본질적 한계
